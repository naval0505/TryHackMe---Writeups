# Warzone 2 - TryHackMe Writeup

## Overview

Today we are going to solve another **Defensive Security** challenge from **TryHackMe** named **Warzone 2**.

Unlike offensive security rooms where the objective is to compromise a target, this room focuses entirely on **Network Traffic Analysis**, **Incident Response**, and **Threat Detection**. We are given a packet capture (PCAP) file generated after multiple security alerts were triggered within an enterprise environment. Our task is to investigate the captured traffic, determine whether the alerts represent a true positive, recover the downloaded malware, identify Indicators of Compromise (IOCs), and validate the findings using threat intelligence platforms.

Throughout this investigation we will use **Brim**, **Wireshark**, **NetworkMiner**, and **VirusTotal** to reconstruct the attack.

---

# Scenario

The provided scenario states:

> You work as a Tier 1 (L1) Security Analyst for a Managed Security Service Provider (MSSP). During routine monitoring several IDS alerts were triggered including:
>
> - Misc Activity
> - A Network Trojan Was Detected
> - Potential Corporate Privacy Violation
>
> Your responsibility is to investigate the packet capture, determine whether the alerts are legitimate, recover the malware artifacts, and identify all indicators associated with the attack.

---

# Investigation Methodology

The investigation follows a structured incident response methodology.

1. Review IDS alerts generated by Brim.
2. Validate malicious communications.
3. Identify the suspicious source and destination IP addresses.
4. Recover the downloaded malware.
5. Inspect HTTP requests.
6. Extract Indicators of Compromise.
7. Validate infrastructure using VirusTotal.
8. Build an attack timeline.

---

# Tools Used

- Brim
- Wireshark
- NetworkMiner
- VirusTotal

Each tool serves a different purpose throughout the investigation.

**Brim**

Used for quickly querying Zeek logs and reviewing IDS alerts generated by Suricata.

**Wireshark**

Used for packet-level inspection including HTTP requests, TCP streams, and protocol analysis.

**NetworkMiner**

Used to reconstruct transferred files and recover malware payloads from captured traffic.

**VirusTotal**

Used to validate domains, IP addresses, malware reputation, and infrastructure associated with the attack.

# Opening the Packet Capture

The investigation begins by opening the supplied PCAP inside **Brim**.

Since the challenge already mentions that IDS alerts were generated, Brim provides the fastest method for locating them.

Rather than manually inspecting thousands of packets, Brim indexes Zeek logs allowing extremely fast searching.

---

# Question 1 – Network Trojan Alert

The first question asks for the IDS signature corresponding to:

> **A Network Trojan Was Detected**

Inside Brim we search for the keyword:

```text
trojan
```

Only one matching alert appears.

Opening the event reveals the complete Suricata signature.

Recovered Signature:

```text
ET MALWARE Likely Evil EXE download from MSXMLHTTP non-exe extension M2
```

This alert indicates that a Windows system downloaded an executable while attempting to disguise it using a non-standard file extension.

Attackers frequently rename executable payloads to evade simple detection rules and user suspicion.

---

# Question 2 – Corporate Privacy Violation Alert

The second alert references:

> Potential Corporate Privacy Violation

Again, Brim allows us to locate the event instantly.

Search query:

```text
Potential Corporate
```

Only one alert matches.

Recovered Signature:

```text
ET POLICY PE EXE or DLL Windows file download HTTP
```

Unlike the previous alert, this signature specifically detects Windows executable or DLL downloads transferred over HTTP.

Because enterprise environments generally restrict direct executable downloads from unknown hosts, this behavior is considered suspicious and often indicates malware delivery.

---

Opening either IDS alert reveals the complete network metadata.

Among the fields is the external IP responsible for triggering the detection.

Recovered Address:

```text

```

The challenge requests the answer in defanged format.

Final IOC:

```text
185[.]118[.]164[.]8
```

This IP address becomes the primary malicious server investigated throughout the remainder of the analysis.

# Question 4 – Recovering the Malicious Download

After identifying the malicious server, the next objective is determining exactly what file the victim downloaded.

For this stage we switch from Brim to **Wireshark** because packet-level analysis provides complete HTTP request details.

Inspecting the first HTTP request reveals:

```http
GET /czwih/fxla.php?l=gap1.cab HTTP/1.1
```

Host Header:

```text
awh93dhkylps5ulnq-be.com
```

Complete Request URI:

```text
http://awh93dhkylps5ulnq-be.com/czwih/fxla.php?l=gap1.cab
```

The request downloads a Cabinet archive named:

```text
gap1.cab
```

Rather than requesting an executable directly, the attacker packages the malware inside a Microsoft Cabinet archive.

Using compressed archives is a common malware delivery technique because it can bypass email filters, antivirus signatures, and content inspection systems.

The challenge requires the URI in defanged format.

Recovered IOC:

```text
awh93dhkylps5ulnq-be[.]com/czwih/fxla[.]php?l=gap1[.]cab
```

# Question 5 – Extracting the Payload

The downloaded object is not the final malware.

Instead, the CAB archive contains another file.

To recover transferred files more efficiently we open the PCAP inside **NetworkMiner**.

Unlike Wireshark, NetworkMiner automatically reconstructs files transferred across network sessions.

Navigating to the **Files** tab displays the extracted contents of the Cabinet archive.

Recovered Payload:

```text
draw.dll
```

Instead of downloading an executable directly, the attacker delivers a Dynamic Link Library (DLL).

DLL-based malware is frequently used by malware loaders because it can later be executed using tools such as:

- rundll32.exe
- regsvr32.exe
- custom loaders
- reflective DLL injection

This technique often bypasses traditional security controls that focus primarily on executable files.

---

# Question 6 – User-Agent Analysis

Returning to the HTTP request inside Wireshark provides another valuable indicator.

The malware identifies itself using the following User-Agent string.

```text
Mozilla/4.0 (compatible; MSIE 7.0; Windows NT 10.0; WOW64; Trident/8.0; .NET4.0C; .NET4.0E)
```

Although this appears to resemble Internet Explorer, the combination is unusual.

Several inconsistencies immediately stand out.

- Internet Explorer 7 normally does not run on Windows 10.
- The Trident rendering engine version does not match IE7.
- The browser attempts to masquerade as a legitimate Microsoft browser.

Malware authors commonly spoof User-Agent strings in an attempt to blend malicious HTTP traffic with normal enterprise web browsing activity.

# VirusTotal Correlation

With the initial Indicators of Compromise identified, the next step is validating the malicious infrastructure using **VirusTotal**. Correlating packet captures with external threat intelligence allows us to determine whether the contacted domains and IP addresses are already associated with known malware campaigns.

Rather than relying solely on the IDS alerts, VirusTotal helps confirm whether the observed infrastructure has previously been reported by security researchers and antivirus vendors.

# Threat Intelligence Analysis

VirusTotal shows multiple security vendors classifying these domains as malicious.

The infrastructure has previously been associated with malware distribution campaigns, demonstrating why reputation-based intelligence is an important supplement to packet analysis.

Instead of interacting with live malware infrastructure, analysts can safely validate domains through public threat intelligence databases.

# Question 8 – IP Addresses Classified as Not Suspicious Traffic

While reviewing the VirusTotal information another interesting observation appears.

Some IP addresses contacted during the investigation are marked as **Not Suspicious Traffic**, despite hosting domains that later become associated with malware.

Examining the contacted domains reveals:

```text

knockoutlights.com

```

Further investigation of these IP addresses identifies two servers marked as Not Suspicious.

Recovered IP Addresses:

```text
64.225.65.166

142.93.211.176
```

The challenge requests the answer in numerical order and defanged format.

Final IOC:

```text
64[.]225[.]65[.]166,142[.]93[.]211[.]176
```

This demonstrates an important concept during incident response.

IP reputation constantly changes over time.

A server may not currently appear malicious while still hosting domains that participated in previous malware campaigns.

Therefore analysts should investigate both domains and IP addresses rather than relying solely on reputation scores.

# Detection Opportunities

Several opportunities existed to detect or prevent this attack before execution.

- IDS signatures successfully detected executable downloads over HTTP.
- HTTP traffic to suspicious domains generated Suricata alerts.
- The downloaded Cabinet archive could have been inspected by secure web gateways.
- Reputation-based filtering could have blocked known malicious domains.
- Endpoint Detection and Response (EDR) solutions could have detected the DLL execution.
- DNS monitoring could have identified communication with previously reported malicious infrastructure.

# Lessons Learned

This investigation demonstrates how multiple forensic tools complement one another during incident response. Brim allows analysts to rapidly identify IDS alerts without manually reviewing every packet, Wireshark provides detailed protocol analysis, NetworkMiner simplifies malware extraction from captured traffic, and VirusTotal adds valuable threat intelligence by correlating observed infrastructure with previously reported campaigns. Even infrastructure currently classified as "Not Suspicious" may still be linked to malicious domains through historical DNS records, emphasizing the importance of combining network analysis with external intelligence sources.

---

# Conclusion

The **Warzone 2** challenge provides a practical introduction to network-based incident response using real-world forensic tools. Starting from IDS alerts, we confirmed that the event represented a true malware infection, identified the malicious server responsible for delivering the payload, recovered the downloaded Cabinet archive, extracted the embedded DLL, analyzed HTTP communications, and validated all recovered infrastructure using VirusTotal. The investigation highlights how packet captures, IDS telemetry, file reconstruction, and threat intelligence work together to reconstruct an attack and produce actionable Indicators of Compromise that can be used to improve detection and future incident response.
