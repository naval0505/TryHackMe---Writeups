# TryHackMe - Investigating with Kibana

> **Category:** Defensive Security | SIEM | Network Investigation
>
> **Difficulty:** Easy
>
> **Objective:** Investigate HTTP connection logs within Kibana, identify suspicious outbound communication, trace Command and Control (C2) traffic, and recover the malicious file referenced by the attacker.

---

# Scenario

During routine Security Operations Center (SOC) monitoring, Analyst **John** observed an alert generated by the organization's Intrusion Detection System (IDS). The alert indicated possible Command and Control (C2) communication originating from a workstation belonging to **Browne**, a user from the Human Resources department.

Further investigation suggested that the compromised host had accessed a suspicious file containing a malicious pattern beginning with:

```text
THM{__________}
```

Because of limited forensic resources, only **one week of HTTP connection logs** was collected and ingested into the **connection_logs** index inside **Kibana**.

Our objective is to investigate these logs, identify the compromised system, trace the attacker’s infrastructure, recover the downloaded file, and answer the challenge questions.

# Initial Investigation

The investigation began by adjusting the Kibana time filter to include the **entire month of March 2022**.

This ensures that every available event within the investigation period is visible before narrowing the search.

---

# Question 1

## How many events were returned for the month of March 2022?

After selecting the complete date range covering March 2022, Kibana displayed:

```text
1482 Events
```

---

## Analysis

Establishing the total number of events provides an understanding of the available dataset before applying additional filters.

This also confirms that the required investigation timeframe has been selected correctly.

---

# Question 2

## What IP address belongs to the suspected user?

Filtering the HTTP logs associated with the suspicious activity revealed the source workstation.

Recovered IP:

```text
192.166.65.54
```

---

## Analysis

The **source IP address** identifies the workstation initiating the suspicious outbound connections.

This becomes the primary focus for the remainder of the investigation.

---

# Question 3

## Which Windows binary was used to download the malicious file?

Reviewing the HTTP requests generated by the compromised workstation revealed the following event.

```text
Timestamp:

Mar 10, 2022

Destination:

104.23.99.190

Host:

pastebin.com

Method:

HEAD

URI:

/yTg0Ah6a

User-Agent:

bitsadmin
```

---

## Analysis

The HTTP **User-Agent** identifies the application responsible for generating the request.

Recovered User-Agent:

```text
bitsadmin
```

**BITSAdmin** is a legitimate Microsoft Windows utility used for Background Intelligent Transfer Service (BITS) operations.

Attackers frequently abuse BITSAdmin because it can:

- Download remote files
- Transfer data quietly
- Blend into legitimate Windows activity
- Evade basic detection mechanisms

The use of a legitimate Windows binary is an example of a **Living Off The Land (LOLBins)** technique.

---

# Question 4

## Which file-sharing service acted as the Command and Control server?

The HTTP request clearly identifies the destination host.

```text
Host:

pastebin.com
```

---

## Analysis

Although **Pastebin** is a legitimate text-sharing platform, threat actors frequently misuse it as lightweight Command and Control infrastructure.

Common attacker uses include:

- Hosting commands
- Delivering payload URLs
- Storing encoded malware configuration
- Delivering PowerShell commands

Using legitimate services helps malicious traffic blend into normal network activity.

---

# Question 5

## What is the complete C2 URL?

Combining the host and requested URI reveals the full destination.

Host:

```text
pastebin.com
```

URI:

```text
/yTg0Ah6a
```

Complete URL:

```text
https://pastebin.com/yTg0Ah6a
```

---

## Analysis

The URI identifies the exact Pastebin resource contacted by the infected host.

This represents the malware's Command and Control location and is an important **Indicator of Compromise (IOC)**.

During an incident response investigation, URLs such as this should be:

- Blocked
- Added to detection rules
- Shared with threat intelligence platforms

---

# Question 6

## Which file was accessed?

Opening the recovered Pastebin page revealed its contents.

The referenced file:

```text
secret.txt
```

The page also contained the challenge flag.

```text
THM{SECRET__CODE}
```

---

## Analysis

The malware contacted the Pastebin page to retrieve instructions pointing toward the file **secret.txt**.

Although this challenge simplifies the investigation, in real-world attacks similar pages often contain:

- PowerShell commands
- Download URLs
- Malware configuration
- Encryption keys
- Additional payload locations

# Key Learning Points

- SIEM platforms such as Kibana allow analysts to efficiently filter and investigate large volumes of log data.
- Selecting the correct time range is an essential first step in any investigation.
- Legitimate Windows utilities like **BITSAdmin** are frequently abused by attackers to download malicious content while blending into normal system activity.
- Public services such as **Pastebin** are commonly used as lightweight Command and Control infrastructure because they appear trustworthy and are often allowed through corporate firewalls.
- User-Agent strings, destination hosts, requested URIs, and full URLs all serve as valuable Indicators of Compromise that can be used for detection, threat hunting, and incident response.

---

# Conclusion

This challenge demonstrates a practical workflow for investigating suspicious HTTP traffic using Kibana. Beginning with an IDS alert, the investigation focused on narrowing the available log data, identifying the compromised workstation, and tracing its outbound communications. Analysis of the HTTP request revealed that the attacker leveraged the legitimate Windows utility **BITSAdmin** to communicate with a Pastebin-hosted Command and Control endpoint.

By reconstructing the full URL and examining the remote content, the referenced file and challenge flag were successfully recovered. The exercise highlights the importance of analyzing HTTP metadata—including source addresses, User-Agent strings, destination hosts, and requested resources—to identify attacker infrastructure and build meaningful Indicators of Compromise during real-world incident response investigations.
